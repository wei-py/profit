# Fix Tauri fs.write_file Permission Denied

## Root Cause Analysis

After researching Tauri 2 plugin-fs v2.5.0 permission model, the issue has two layers:

### Layer 1: Wrong command permission identifier

In Tauri 2 plugin-fs v2.x, the `writeFile` function maps to the command permission `fs:allow-write-file` (NOT `fs:allow-write`). The current capability has `fs:allow-write` which is a legacy/v1-style generic permission that doesn't match the actual command in v2.

The error message lists ALL permissions "associated with this command":
- `fs:allow-write-file` ← the specific permission for `writeFile()`
- `fs:allow-home-write`, `fs:allow-home-write-recursive` ← combined scope+command for home dir
- `fs:allow-document-write`, `fs:allow-document-write-recursive` ← for documents dir
- `fs:write-all`, `fs:write-files` ← catch-all permissions

### Layer 2: Missing scope for non-app directories

Even with the correct command permission, Tauri 2 scopes restrict WHICH paths can be written to. The `fs:default` permission set only allows read/write within app-specific directories ($APPDATA, $APPCONFIG, etc.). User-chosen paths from `dialog:save()` are NOT in these default scopes.

The earlier attempts using `"scope": { "fs": { "allow": [...] } }` and `fs:scope` as a permission identifier both failed because:
1. `fs:scope` as a top-level capability scope DOES work, BUT
2. The command permission `fs:allow-write` doesn't map to `writeFile` in v2.5.0

## Fix: Update capabilities/default.json

The fix requires BOTH the correct command permission AND scope coverage for user directories.

### Correct approach: Use combined scope+command permissions

Tauri 2 plugin-fs v2 generates combined permissions for each base directory like:
- `fs:allow-home-write-recursive` — enables `writeFile` + scope for $HOME/**
- `fs:allow-desktop-write-recursive` — enables `writeFile` + scope for $DESKTOP/**
- etc.

These permissions INLINE both the command authorization and the scope definition.

### Final capability file

```json
{
  "$schema": "../gen/schemas/desktop-schema.json",
  "identifier": "default",
  "description": "enables the default permissions and full filesystem access for config files",
  "windows": ["main"],
  "permissions": [
    "core:default",
    "dialog:default",
    "dialog:allow-open",
    "dialog:allow-save",
    "fs:default",
    "fs:allow-exists",
    "fs:allow-home-read-recursive",
    "fs:allow-home-write-recursive",
    "fs:allow-desktop-read-recursive",
    "fs:allow-desktop-write-recursive",
    "fs:allow-document-read-recursive",
    "fs:allow-document-write-recursive",
    "fs:allow-download-read-recursive",
    "fs:allow-download-write-recursive",
    "store:default"
  ]
}
```

### Explanation of changes

| Removed | Added | Reason |
|---------|-------|--------|
| `fs:allow-read` | — | Replaced by specific scope+command combos |
| `fs:allow-write` | — | Doesn't map to `writeFile` in v2.5.0 |
| — | `fs:allow-home-read-recursive` | Read files in $HOME/** |
| — | `fs:allow-home-write-recursive` | Write files in $HOME/** |
| — | `fs:allow-desktop-read-recursive` | Read files on desktop |
| — | `fs:allow-desktop-write-recursive` | Write files on desktop |
| — | `fs:allow-document-read-recursive` | Read files in Documents |
| — | `fs:allow-document-write-recursive` | Write files in Documents |
| — | `fs:allow-download-read-recursive` | Read files in Downloads |
| — | `fs:allow-download-write-recursive` | Write files in Downloads |

The `*recursive` variant is needed because:
- Non-recursive (`fs:allow-home-write`) only allows writing files at the top level of $HOME, not in subdirectories like `$HOME/Documents/...`
- Recursive (`fs:allow-home-write-recursive`) allows writing files in ALL subdirectories

These combined permissions (`fs:allow-<dir>-<action>-recursive`) bundle both the command authorization and scope definition into a single permission identifier, which is the intended usage pattern in Tauri 2 plugin-fs v2.

## Files to Modify

| File | Change |
|------|--------|
| `src-tauri/capabilities/default.json` | Replace `fs:allow-read`/`fs:allow-write` with combined scope+command permissions for home/desktop/document/download directories |

## Verification

After applying the changes, run:
```bash
pnpm tauri dev
```

Then test: Open a config Excel, make changes, click "保存配置". The save should succeed without the `fs.write_file not allowed` error.

## Caveats

- The `$HOME/**` scope covers the ENTIRE home directory, including sensitive files. For a production/distributed app, this is too permissive. For a local dev/config tool, it's acceptable.
- On macOS, `$HOME` resolves to `/Users/<username>/` which covers Documents, Desktop, Downloads, etc.
- If stricter scope is needed later, each `fs:allow-*-recursive` permission can be optionally constrained by adding an `allow` array of specific path patterns, but the combined permissions already handle typical use cases.
