# Bug Fix Plan — 3 Runtime Bugs

## Bug #1: TemplateModal.vue — confirmAction.value() ✅ DONE

**Problem**: `@ok="confirmAction(); showConfirmModal = false"` — `confirmAction` is `ref(() => {})`, and in Vue 3 `<script setup>` templates, refs are auto-unwrapped, BUT the inline expression with multiple statements can cause ambiguity. Safer to use a named handler function with explicit `.value`.

**Actual Fix Applied**:

- Added `onConfirmOk()` function with explicit `.value` calls
- Replaced `@ok="confirmAction(); showConfirmModal = false"` with `@ok="onConfirmOk"`

---

## Bug #2: ReverseCalcModal.vue — 反推计算功能失效 ✅ DONE

**Problem**: Multiple issues causing reverse calculation to fail:

1. Watch only watched `props.open`, missing `skuIndex` updates
2. `v-model` + `@input` on `<input type="number">` in Vue 3.5 — inline `@input` handler reading ref value could get stale data due to v-model handler ordering ambiguity
3. `recalc` (formerly `onCalcFieldChange`) overwrote the field the user was actively typing in, resetting it to "0.00" on every keystroke
4. Binary search range `hi = cost * 20` invalid when cost is 0 or small (hi ≤ lo)

**Actual Fix Applied**:

1. Watch both `props.open` + `props.skuIndex` together, with `skuIdx < 0` guard
2. Replaced `onCalcFieldChange(field)` (reading from ref) with `recalc(field, event)` reading value from `event.target.value` directly — bypasses v-model timing issues
3. Template: `@input="onCalcFieldChange('price')"` → `@input="(e) => recalc('price', e)`"
4. **Don't overwrite the field the user is editing**:
   - 'price' branch: only updates calcProfit + calcMargin (not calcPrice)
   - 'profit' branch: only updates calcPrice + calcMargin (not calcProfit)
   - 'margin' branch: only updates calcPrice + calcProfit (not calcMargin)
5. Binary search range guard: `if (hi <= lo) hi = lo + 100`
6. Early return on `raw <= 0` for profit/margin targets
7. `skuIndex < 0` guards in `runEngine()` and `recalc()`

## Bug #3: CountryPage.vue — 删除按钮调错函数 ✅ DONE

**Problem**: Three drag lists (字段/选项组/模板) all had ✕ buttons calling `deleteRow()` (which deletes a country row by id). Should have dedicated delete functions for each list type.

**Actual Fix Applied**:

Added 3 dedicated delete functions:

```js
function deleteField(idx) — removes from store['计算字段'] + localFields
function deleteOpt(idx) — removes from store['选项组'] + store['选项值'] + localOptGroups
function deleteTpl(idx) — removes from store['计算模板'] + store['费用规则'] + localTemplates
```

Template buttons updated:
- `@click="deleteRow(f.字段键)"` → `@click="deleteField(i)`
- `@click="deleteRow(g.编号)"` → `@click="deleteOpt(i)`
- `@click="deleteRow(t.编号)"` → `@click="deleteTpl(i)"
