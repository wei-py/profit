# Bug Fix Plan — 3 Runtime Bugs

## Bug #1: TemplateModal.vue — confirmAction.value() (line 218)

**Problem**: `@ok="confirmAction(); showConfirmModal = false"` — `confirmAction` is `ref(() => {})`, and in Vue 3 `<script setup>` templates, refs are auto-unwrapped, BUT the inline expression with multiple statements can cause ambiguity. Safer to use a named handler function with explicit `.value`.

**Fix**:

- Add `onConfirmOk()` function in `<script setup>` that calls `confirmAction.value()` and `showConfirmModal.value = false`
- Replace `@ok="confirmAction(); showConfirmModal = false"` with `@ok="onConfirmOk"`

```js
function onConfirmOk() {
  confirmAction.value();
  showConfirmModal.value = false;
}
```

Template change (line 218):

```
<ConfirmModal ... @ok="onConfirmOk" @close="showConfirmModal = false" />
```

**Note**: The two-script-block issue is already fixed (current file has single `<script setup>`).

## Bug #2: ReverseCalcModal.vue — 反推计算功能失效

**Problem**: The watch only watches `props.open`. If `props.skuIndex` hasn't been updated when the watch fires, `createStore.skus[props.skuIndex]` could be undefined, causing early return and no initialization. Also, `@input` event + `v-model` interaction could have subtle timing issues.

**Fix**:

1. Change watch to watch BOTH `props.open` AND `props.skuIndex` together, only initialize when `open` is true AND `skuIndex >= 0`
2. Use `nextTick` in the watch to ensure state is fully flushed before reading

```js
watch(
  [() => props.open, () => props.skuIndex],
  ([openVal, skuIdx]) => {
    if (!openVal || skuIdx < 0) return;
    const sku = createStore.skus[skuIdx];
    if (!sku) return;
    const price = Number.parseFloat(sku.inputs["售价"]) || 0;
    const pf = sku.results["净利润"] !== undefined ? Number.parseFloat(sku.results["净利润"]) : 0;
    calcPrice.value = price || "";
    calcProfit.value = pf || "";
    calcMargin.value =
      sku.results["利润率"] !== undefined
        ? (Number.parseFloat(sku.results["利润率"]) * 100).toFixed(2)
        : "";
  },
  { immediate: false },
);
```

Also in `runEngine()`, add a guard for `skuIndex < 0`:

```js
function runEngine(price) {
  if (props.skuIndex < 0) return { profit: 0, margin: 0 }
  const sku = createStore.skus[props.skuIndex]
  ...
}
```

Same guard in `onCalcFieldChange()`:

```js
function onCalcFieldChange(field) {
  if (props.skuIndex < 0) return
  ...
}
```

## Bug #3: CountryPage.vue — 删除按钮调错函数

**Problem**: Three drag lists (字段/选项组/模板) all have ✕ buttons calling `deleteRow()` (which deletes a country row by id). Should have dedicated delete functions for each list type.

**Fix**: Add 3 dedicated delete functions and update template ✕ buttons:

```js
function deleteField(idx) {
  const fields = store.getFieldsByCountry(cpId.value);
  const x = store["计算字段"].indexOf(fields[idx]);
  if (x !== -1) store["计算字段"].splice(x, 1);
  localFields.value.splice(idx, 1);
}

function deleteOpt(idx) {
  const groups = store.getOptionGroupsByCountry(cpId.value);
  const g = groups[idx];
  store["选项组"] = store["选项组"].filter((r) => r.编号 !== g.编号);
  store["选项值"] = store["选项值"].filter((r) => r.所属分组 !== g.编号);
  localOptGroups.value.splice(idx, 1);
}

function deleteTpl(idx) {
  const templates = store.getTemplatesByCountry(cpId.value);
  const t = templates[idx];
  store["计算模板"] = store["计算模板"].filter((r) => r.编号 !== t.编号);
  store["费用规则"] = store["费用规则"].filter((r) => r.所属模板 !== t.编号);
  localTemplates.value.splice(idx, 1);
}
```

Template changes:

- Line 228: `@click="deleteRow(f.字段键)"` → `@click="deleteField(i)"`
- Line 252: `@click="deleteRow(g.编号)"` → `@click="deleteOpt(i)"`
- Line 276: `@click="deleteRow(t.编号)"` → `@click="deleteTpl(i)"`
