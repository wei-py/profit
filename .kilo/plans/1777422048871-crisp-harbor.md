# Fix PresetPage Save & Data Persistence Issues

## Root Cause Analysis

After reviewing the git diff, all JSDoc changes are comment-only (no logic modifications). The `CACHE_GROUP_KEY` in OptionPage was already restored. The issues described are likely pre-existing bugs exposed by testing:

### Issue 1: Keep Config — Save Not Working / No User Feedback

The `saveConfigExcel()` async function in `useFileIO.js` is called via `@click="saveConfigExcel"` in the PresetPage template without any error handling or user feedback. If `writeFile` fails silently (file locked, permissions, path invalid), the user sees no indication.

Additionally, the rule action serialization in `excel-writer.js` has a **camelCase/snake_case mismatch** — the action objects use camelCase (`configJson`, `actionId`, `actionType`, `targetField`) but the writer headers reference snake_case props (`config_json`, `action_id`, etc), causing action data to be lost when saving rules.

### Issue 2: Data Lost After Refresh

The `restoreLastPath()` function in `useFileIO.js` relies on Tauri's store plugin. If the store is unavailable or the saved path is invalid, it fails silently. There's no fallback or user notification.

**File:** `src/services/excel-writer.js`

Change the action column headers from snake_case to camelCase to match how actions are actually stored:

```diff

-  { key: 'action_id', prop: 'action_id' },
-  { key: 'rule_id', prop: 'rule_id' },
+  { key: 'action_id', prop: 'actionId' },
+  { key: 'rule_id', prop: 'ruleId' },
   { key: 'sort', prop: 'sort' },
-  { key: 'action_type', prop: 'action_type' },
-  { key: 'target_field', prop: 'target_field' },
-  { key: 'config_json', prop: 'config_json' },
+  { key: 'action_type', prop: 'actionType' },

 ]
```
